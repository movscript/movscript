# MovScript V2 用户视角降噪与画布卡片设计

本文档是 `movscript-v2-product-design.md` 的体验设计补充稿。

V2 的底层模型需要清晰边界，但用户界面不能把所有模型概念直接暴露出来。产品真正要解决的问题是：

```text
用户给系统一份剧本、文案或分镜脚本后，能快速看到雏形，知道哪里需要确认、哪里缺素材、哪里可以继续生成。
```

因此，界面设计的重点不是展示系统有多少对象，而是帮助用户持续回答五个问题：

```text
这是什么？
系统理解得对不对？
现在缺什么？
我能做什么决定？
决定后会保存到哪里？
```

## 1. 设计判断

当前 V2 的底层分层是合理的，但它应该主要服务工程实现：

```text
ScriptSection
Situation
StoryboardLine
ContentUnit
Keyframe
CreativeReference
CreativeReferenceState
AssetRequirement
ContentVersion
WorkItem
```

但这些不应该全部成为用户一眼看到的产品语言。

如果 UI 直接展示这些对象名，MovScript 会显得像一个复杂的制片数据系统，而不是一个 AI 影视工作台。用户会被迫理解系统内部如何建模，反而削弱“导入剧本就能看到片子”的核心价值。

所以 V2 需要两套语言。用户默认只看到第一套：

```text
用户语言：剧本段、情境、画面、关键画面、素材缺口、候选视频、待处理
系统语言：ScriptSection、Situation、ContentUnit、Keyframe、AssetRequirement、ContentVersion、WorkItem
```

用户语言负责界面体验。系统语言只在管理后台、调试模式、API、审计和开发协作中出现。

## 2. 用户视角降噪原则

### 2.1 一级界面只保留五类对象

用户在主流程里只需要理解：

```text
剧本
情境
画面
素材
结果
```

更具体一点：

| 用户看到 | 底层对象 | 用户心智 |
| --- | --- | --- |
| 剧本段 | ScriptSection | 这一段原文在讲什么 |
| 情境 | Situation | 这里发生了什么 |
| 画面 | ContentUnit / StoryboardLine | 这一段要被拍成什么画面 |
| 关键画面 | Keyframe | 当前预演看到的视觉锚点 |
| 素材缺口 | AssetRequirement | 继续生成还缺什么 |
| 候选视频 | ContentVersion | 这个片段能不能采用 |
| 待处理 | WorkItem | 有一件事正在做或等人处理 |

不建议在主界面直接展示：

```text
ContentUnit
CreativeReferenceState
AssetRequirement
ContentVersion
WorkItem
```

这些可以出现在：

- 管理后台
- 右侧详情的调试区域
- API 文档
- 导出数据
- 审计记录
- 开发者模式

### 2.2 卡片不展示完整字段，只展示当前决策

卡片不是数据库记录详情页。

一张卡片应该只承载：

```text
身份：这是什么
摘要：系统如何理解它
状态：当前能不能继续
缺口：还缺什么
动作：用户现在能做什么
落点：动作结果保存到哪里
```

完整字段、版本历史、审计信息、权限、底层 ID、长文本 prompt 都不应该挤在卡片上。它们应该进入右侧详情面板或二级弹窗。

### 2.3 用户不应该先理解任务系统

`WorkItem` 是执行层对象，不是创作事实源。

用户在主流程里不需要看到“WorkItem”这个词，只需要看到：

```text
正在生成
待确认
需要补素材
需要重做
失败，可重试
已完成，待采用
```

任务完成只说明执行完成，不代表结果已经被采用。

例如：

```text
AI 视频生成完成
```

在用户界面里应该继续要求一个采用决策：

```text
采用为当前片段
保留为候选
重新生成
放弃
```

### 2.4 画布不是主流程入口

V2 的主入口应该是“剧本预演”，不是“画布”。

画布应该从具体对象打开：

```text
从一个情境打开画布
从一个画面打开画布
从一个关键画面打开画布
从一个素材缺口打开画布
从一个候选视频打开画布
```

用户打开画布时，系统应该已经知道下面这些信息，但不需要把这些词展示给用户：

```text
这是从哪个画面/素材/结果点进来的
生成出来以后默认放回哪里
当前可以读取哪些剧本、人物和素材信息
用户现在能做哪些动作
```

不要鼓励用户从全项目空白画布开始。空白画布容易制造无落点输出，也会把产品体验拉回通用节点工具。

### 2.5 先保证一条路径好用

第一版不应该追求覆盖所有对象和所有生产方式。

最重要的路径是：

```text
导入剧本
-> 生成预演
-> 用户点开一个不满意的画面
-> 修改情境或画面描述
-> 生成关键画面
-> 采用关键画面
-> 继续生成候选视频
```

这条路径如果顺，用户会觉得产品有价值。

这条路径如果不顺，即使模型再完整，产品也会显得难用。

## 3. 产品语言重映射

### 3.1 导航命名

推荐保留 V2 原导航，但实际页面标题要更口语：

| 导航 | 页面标题建议 | 页面目的 |
| --- | --- | --- |
| 项目首页 | 今天要处理什么 | 推动项目继续往前走 |
| 剧本预演 | 先看整片雏形 | 导入剧本并播放预演 |
| 创作资料 | 角色、地点和风格 | 确认 AI 对资料的理解 |
| 素材准备 | 还缺哪些素材 | 管理缺失、候选、已锁定 |
| 内容生产 | 把画面做成片段 | 关键画面升级成视频 |
| 制作任务 | 待处理事项 | 执行、分配、审核、返工 |
| 交付 | 检查并导出 | 时间线、缺失检查、版本 |
| 画布 | 创作画布管理 | 最近画布、模板、历史运行 |

### 3.2 对象命名

产品界面优先使用以下命名：

| 产品名 | 系统名 | 说明 |
| --- | --- | --- |
| 剧本段 | ScriptSection | 原文切分后的语义段 |
| 情境 | Situation | AI 对发生条件和上下文的理解 |
| 画面 | ContentUnit | 会出现在预演时间线里的单位 |
| 分镜稿 | StoryboardLine | 用户和 AI 共同编辑的结构化表达 |
| 关键画面 | Keyframe | 视觉锚点或候选图 |
| 创作资料 | CreativeReference | 人物、地点、道具、产品、风格 |
| 资料状态 | CreativeReferenceState | 特定情境里的表现状态 |
| 素材缺口 | AssetRequirement | 为生产某个画面还缺的输入 |
| 候选视频 | ContentVersion | 生产出来但未必采用的视频版本 |
| 待处理事项 | WorkItem | 执行层工作 |

产品界面里可以在高级模式显示系统名，例如：

```text
画面 CU-03
情境 Situation #12
候选视频 ContentVersion #5
```

但默认不显示。

## 4. 重新设计画布卡片

画布卡片应该按用户决策分成六类，而不是按数据库对象一一铺开。

推荐第一版支持用户能理解的六类卡片：

```text
这一段发生什么
预演里的一个画面
关键图候选
还缺一个素材
人物当前样子
生成的视频候选
```

后续再补：

```text
剧本段卡
分镜稿卡
执行卡
工具卡
候选组卡
交付检查卡
```

### 4.1 “这一段发生什么”卡

用户问题：

```text
系统对这一段剧情理解得对吗？
```

界面不要叫“情境卡”，可以叫：

```text
这一段发生什么
系统理解
剧情理解
```

卡片应该展示：

- 一句话描述
- 来自第几段剧本
- 包含哪些人和物
- 当前确认状态
- 下一步会生成画面

主要动作：

- 改一下理解
- 确认理解

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧打开“这一段发生什么”的详情，相关画面高亮 | 选中当前剧情理解对象 |
| 改一下理解 | 右侧出现自然语言编辑框，用户直接改描述 | 保存新的理解版本，标记下游画面可能需要更新 |
| 确认理解 | 卡片状态变成“已确认”，下一步提示生成画面 | 这个理解可以被用于生成预演画面 |

不应该展示给普通用户：

- 长 prompt
- 原始 JSON
- 完整关系图
- 所有字段
- `Situation`
- `owner_type`
- `事实源`

卡片示例：

```text
这一段发生什么
雨夜老城区窄巷，林夏攥着旧伞，与顾言保持距离对峙。

来自：剧本段 03
涉及：林夏 / 顾言 / 旧伞 / 老城区巷口
状态：待确认

[改一下理解] [确认理解]
确认后：用它生成下面的预演画面
```

### 4.2 “预演里的一个画面”卡

用户问题：

```text
这一段预演里具体看到什么？
```

卡片应该展示：

- 画面标题
- 画面描述
- 顺序和时长
- 当前关键图
- 生产状态
- 是否缺素材

主要动作：

- 改画面
- 生成关键图
- 做成视频

高级动作可以藏到更多菜单：

- 拆成两个画面
- 合并到上一个画面
- 调整顺序

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧显示画面描述、时长、当前关键图和缺失素材 | 选中当前预演画面 |
| 改画面 | 打开画面编辑器，用户改画面、景别、运动和时长 | 只更新这个预演画面的描述，不改剧本原文 |
| 生成关键图 | 显示生成中，完成后新增关键图候选 | 创建一批候选图，等待用户采用 |

卡片示例：

```text
预演里的一个画面 · 2 秒
旧伞伞骨特写，纸条从缝隙滑出，雨水打湿纸面。

当前关键图：v2 已采用
素材：旧伞已锁定，纸条参考缺失
生产：可生成视频

[改画面] [生成关键图] [做成视频]
生成结果会回到：这个预演画面
```

### 4.3 “关键图候选”卡

用户问题：

```text
这张图能不能作为当前画面的视觉锚点？
```

卡片应该展示：

- 图片预览
- 所属画面
- 候选编号
- 生成来源
- 采用状态
- 明显问题

主要动作：

- 采用这张
- 保留候选
- 再生成
- 作为参考继续生成
- 保存为素材

关键边界：

```text
采用关键图 = 当前预演画面换成这张图
不等于改写剧本
不等于改写“这一段发生什么”
```

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧显示大图、所属画面、生成时间、问题标签 | 选中这张候选图 |
| 采用这张 | 时间线里的当前画面缩略图换成这张，卡片变成“已采用” | 把当前画面的视觉锚点改为这张图 |
| 再生成 | 当前候选保留，同时开始生成新候选 | 基于同一画面描述再生成一组图 |

卡片示例：

```text
关键图候选 2/4
纸条滑出旧伞，雨夜冷色，信息清晰。

属于：第 3 个画面
状态：候选

[采用这张] [再生成] [保存为素材]
采用后：第 3 个画面会显示这张图
```

### 4.4 “还缺一个素材”卡

用户问题：

```text
继续生成还缺什么？
```

卡片应该展示：

- 缺的是什么
- 用在哪里
- 当前状态：缺失 / 有候选 / 已使用
- 候选素材数量
- 为什么需要它

主要动作：

- 上传素材
- 从已有素材选择
- AI 生成参考
- 使用这个素材
- 标记不需要

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧显示这个缺口影响哪些画面，以及现有候选 | 选中当前素材需求 |
| 上传素材 | 打开上传弹窗，上传后变成“有候选” | 新素材先作为候选，不自动使用 |
| 生成参考 | 显示生成中，生成后加入候选列表 | 用当前剧情和人物信息生成参考图 |
| 使用这个素材 | 状态变成“已使用”，相关画面的缺素材提醒消失 | 把素材绑定到这个缺口 |

卡片示例：

```text
还缺一个素材
林夏雨夜受伤状态 · 正面半身伤痕参考

用于：情境 08 / 画面 02-04
状态：缺失
原因：生成近景时需要保持伤痕连续性

[上传] [生成参考] [从素材库选择]
选好后：后续近景会使用这个参考
```

### 4.5 “人物当前样子”卡

用户问题：

```text
这个人物/道具/地点在当前情境里是什么状态？
```

卡片应该展示：

- 人物是谁
- 当前样子一句话
- 影响哪些画面
- 哪些特征需要保持一致
- 是否缺参考素材

主要动作：

- 改样子
- 保持一致
- 生成状态参考

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧显示人物当前样子、影响范围和相关素材 | 选中当前人物在这段剧情里的表现 |
| 改样子 | 打开自然语言编辑器，用户改服装、伤痕、情绪 | 更新这段剧情里的人物表现描述 |
| 保持一致 | 卡片变成“已保持”，相关画面生成时都会带上这个约束 | 把这个人物表现作为相关画面的生成约束 |

卡片示例：

```text
林夏 · 雨夜受伤状态
湿透风衣，左颧擦伤，压抑愤怒，攥着旧伞。

作用范围：情境 08 / 画面 02-04
连续性：需要锁定
素材缺口：伤痕参考缺失

[改样子] [保持一致] [生成状态参考]
保持后：接下来 3 个画面都会使用这个样子
```

### 4.6 “生成的视频候选”卡

用户问题：

```text
这个生成出来的视频能不能替换当前预演片段？
```

卡片应该展示：

- 视频预览
- 所属画面
- 来源任务
- 采用状态
- 问题标签
- 与当前版本的关系

主要动作：

- 用这个视频
- 保留为候选
- 返工
- 对比版本
- 放弃

关键边界：

```text
生成完成 != 已采用
用这个视频 = 当前预演画面切换成这个视频
```

点击效果：

| 点击点 | 用户看到什么 | 系统后台做什么 |
| --- | --- | --- |
| 点卡片 | 右侧播放视频，显示问题标签和所属画面 | 选中这个视频候选 |
| 用这个视频 | 时间线第 3 个画面从图变成视频，状态变成“已用视频” | 把该候选设为当前画面的采用版本 |
| 返工 | 打开返工说明，默认带入问题标签 | 基于旧候选和返工说明发起新生成 |
| 放弃 | 卡片收进历史，不再作为推荐候选 | 标记候选为放弃，但保留记录 |

卡片示例：

```text
生成的视频候选 v1
运动自然，但纸条信息不够清晰。

属于：第 3 个画面
状态：候选

[用这个视频] [对比] [返工] [放弃]
采用后：第 3 个画面播放这段视频
```

## 5. 卡片视觉结构

所有画布卡片建议统一为五段：

```text
Header：类型、标题、状态
Preview：图片/视频/文字摘要
Context：来源、所属对象、引用资料
Decision：缺口、风险、采用状态
Actions：当前可执行动作和输出落点
```

紧凑卡片只保留：

```text
类型
标题
一句话摘要
状态
主动作
```

详情卡片可以展开：

```text
上下文
素材
版本
执行记录
输出落点
```

### 5.1 端口设计

画布卡片可以保留输入/输出端口，但端口语义必须用户可理解：

```text
输入：参考、约束、上下文、素材
输出：关键画面、素材、候选视频、待处理事项
```

不要让用户看到过多技术端口：

```text
json
resource
entity_id
workflow value
```

技术端口可以在开发者模式或模板编辑里显示。

### 5.2 颜色和状态

颜色不应该按数据库类型随意分配，而应该帮助用户判断当前状态：

```text
蓝色：理解和上下文
绿色：可继续 / 已采用
黄色：候选 / 待确认
红色：缺失 / 阻塞
紫色：AI 生成 / 创作变化
灰色：历史 / 归档 / 已放弃
```

状态文案应该是用户动作导向：

```text
待确认
可生成
缺素材
生成中
候选
已采用
需返工
已放弃
```

避免：

```text
pending
running
done
failed
candidate_entity
content_version_status
```

## 6. 剧本预演页里的降噪

剧本预演页是 V2 成败关键。

建议布局：

```text
左侧：剧本原文和剧本段
中间：预演播放器和画面列表
右侧：当前选中对象的确认项、素材缺口、下一步动作
底部：时间线
```

用户第一眼应该看到：

```text
预演片是否生成成功
哪些地方待确认
哪些画面缺素材
下一步最值得做什么
```

不要第一眼展示：

```text
完整对象树
所有实体表
任务队列细节
完整画布节点
```

## 7. 画布页里的降噪

画布打开时必须带上下文。

顶部应显示：

```text
正在编辑：画面 03 · 旧伞纸条特写
默认输出：当前画面的关键画面
可用上下文：情境 08、林夏雨夜状态、旧伞素材
```

画布中默认放入：

```text
当前对象卡
必要上下文卡
当前素材缺口卡
一个推荐工具卡
输出落点卡
```

不要让用户打开后面对空白区域。

## 8. 第一版范围

第一版建议只做：

```text
情境卡
画面卡
关键画面卡
素材缺口卡
候选视频卡
AI 工具卡
```

暂缓：

```text
完整剧本结构画布
复杂执行任务图
全项目自由画布
多层嵌套状态编辑
复杂权限卡片
插件市场式工具编排
```

第一版验收目标：

```text
用户能导入一段剧本，生成预演，点开某个画面，知道问题在哪里，生成新关键画面，并明确采用到当前画面。
```

如果这个目标达不到，就不应该继续扩展更多卡片。

## 9. 管理后台 UI 预览建议

管理后台的 UI 预览可以分成两层：

```text
用户视角预览
系统视角预览
```

用户视角预览展示：

- 用户会看到什么文案
- 主流程卡片如何出现
- 状态如何提示下一步
- 输出落点如何表达

系统视角预览展示：

- 这些卡片分别映射到哪些底层对象
- 端口和 owner 如何绑定
- 采用动作会写入哪里
- 任务和结果如何解耦

这样可以避免团队在讨论 UI 时一直回到底层对象命名，也避免工程实现时丢失边界。

## 10. 低学习成本原型

当前管理后台 UI 预览里应新增一个面向用户语言的原型：

```text
V2 画布卡片 -> 预演修改原型
```

这个原型不展示：

```text
事实源
owner_type
ContentUnit
AssetRequirement
WorkItem
端口
节点
```

它只展示用户能理解的流程：

```text
正在修改：第 3 个画面 · 旧伞纸条特写
```

画布中默认出现六张卡：

```text
这一段发生什么
预演里的一个画面
关键图候选
还缺一个素材
人物当前样子
生成的视频候选
```

右侧固定显示：

```text
点击这张卡后，用户看到什么
点击这个按钮后，界面发生什么
系统后台做什么，但不展示给用户
```

### 10.1 原型主屏

```text
┌──────────────────────────────────────────────────────────────┐
│ 正在修改：第 3 个画面 · 旧伞纸条特写                         │
│ 用户看到的是“这个画面怎么改”，所有生成结果默认回到这个画面。 │
├──────────────┬─────────────────────────────┬─────────────────┤
│ 入口说明      │ 卡片工作区                    │ 点击后发生什么    │
│              │                             │                 │
│ 用户从这里进来│ [这一段发生什么] -> [预演画面] │ 当前选中卡片说明  │
│ - 预演画面    │        ↓          ↘           │ 当前按钮效果说明  │
│ - 待确认理解  │ [还缺一个素材]   [关键图候选] │ 可点击点清单      │
│ - 缺素材提醒  │        ↓             ↓        │ 后台才关心        │
│ - 生成结果    │ [人物当前样子] -> [视频候选]  │                 │
└──────────────┴─────────────────────────────┴─────────────────┘
```

### 10.2 点击点清单

| 用户点击 | 右侧应该显示 | 主画布反馈 |
| --- | --- | --- |
| 这一段发生什么 | 解释系统对剧情的理解，以及“改一下理解 / 确认理解”的效果 | 这张卡高亮，相关画面连线高亮 |
| 改一下理解 | 会打开自然语言编辑框；保存后相关画面提示需要更新 | 卡片状态仍是待确认 |
| 确认理解 | 会把状态改成已确认，并允许继续生成画面 | 卡片状态变绿，流向画面卡 |
| 预演里的一个画面 | 显示画面描述、时长、当前关键图和缺素材提醒 | 这张卡高亮 |
| 改画面 | 打开画面编辑器，只改当前画面 | 当前画面显示“已修改，建议重新生成” |
| 生成关键图 | 显示生成中，完成后新增关键图候选 | 关键图候选卡出现或更新 |
| 关键图候选 | 显示大图、所属画面和采用状态 | 候选卡高亮 |
| 采用这张 | 当前预演画面缩略图换成这张图 | 关键图卡变成已采用 |
| 再生成 | 保留当前候选，再生成一组新图 | 候选数量增加 |
| 还缺一个素材 | 显示缺口影响哪些画面和现有候选 | 缺素材卡高亮 |
| 上传素材 | 打开上传弹窗，上传后变成候选 | 缺失状态变成有候选 |
| 生成参考 | 开始生成参考图，完成后加入候选 | 候选素材数量增加 |
| 人物当前样子 | 显示人物当前样子和影响范围 | 人物卡高亮，相关画面高亮 |
| 改样子 | 打开自然语言编辑器 | 相关画面提示受影响 |
| 保持一致 | 接下来相关画面生成都会带上这个约束 | 卡片状态变为已保持 |
| 生成的视频候选 | 右侧播放视频，显示问题和可操作按钮 | 视频候选卡高亮 |
| 用这个视频 | 时间线第 3 个画面切换成视频 | 视频卡变为已用 |
| 返工 | 打开返工说明输入框 | 新视频候选生成中 |

## 11. 内部概念附录

## 11. 具体用例：纸条特写不清楚

这个用例用于说明“卡片”和“工具”之间的关系。

用户场景：

```text
用户播放预演片，发现第 3 个画面“旧伞纸条特写”里纸条看不清。
用户想重新生成关键图，并进一步生成一个可用的视频片段。
```

用户目标：

```text
让纸条信息更清楚。
保持林夏雨夜受伤状态一致。
拿到一个可以采用的视频候选。
```

### 11.1 原型中的卡片和工具

```text
卡片负责上下文、判断和采用。
工具负责生成。
```

卡片：

| 卡片 | 作用 |
| --- | --- |
| 这一段发生什么 | 告诉工具这一段剧情是什么，不让生成跑偏 |
| 预演里的一个画面 | 当前要修改的画面，所有结果默认回到这里 |
| 人物当前样子 | 约束林夏在这几个画面里保持一致 |
| 还缺一个素材 | 提醒伤痕参考缺失，影响生成稳定性 |
| 关键图候选 | 接收关键图生成工具的结果，等待用户采用 |
| 生成的视频候选 | 接收图生视频工具的结果，等待用户选择 |

工具：

| 工具 | 读取 | 生成 | 结果进入 |
| --- | --- | --- | --- |
| 关键图生成 | 画面描述、人物当前样子、旧伞/伤痕参考 | 4 张关键图候选 | 关键图候选卡 |
| 图生视频 | 已采用关键图、动作说明、时长 | 视频候选 | 生成的视频候选卡 |

### 11.2 交互流程

```text
1. 用户从预演时间线点开第 3 个画面。
2. 画布顶部显示：正在修改：第 3 个画面 · 旧伞纸条特写。
3. 系统自动带入“这一段发生什么”“人物当前样子”“还缺一个素材”。
4. 用户看到关键图生成工具，它已经知道要读取哪些卡片。
5. 用户点击“生成关键图”。
6. 工具生成 4 张图，结果进入“关键图候选”。
7. 用户点击“采用这张”。
8. 第 3 个画面的预演缩略图换成这张图。
9. 图生视频工具变为推荐下一步。
10. 用户点击“生成视频”。
11. 视频结果进入“生成的视频候选”。
12. 用户点击“用这个视频”。
13. 第 3 个画面从关键图预演切换成视频片段。
```

### 11.3 工具和卡片的关系规则

工具不应该直接改最终结果。

例如，关键图生成工具完成后，不应该自动替换预演画面。它只新增候选：

```text
工具运行完成 -> 关键图候选出现
用户点击采用 -> 当前预演画面更新
```

视频生成同理：

```text
工具运行完成 -> 视频候选出现
用户点击用这个视频 -> 当前预演片段更新
```

这样用户会清楚：

```text
生成是一回事。
采用是另一回事。
```

### 11.4 每个点击点的效果

| 点击点 | 用户看到什么 | 其它卡片/工具如何变化 |
| --- | --- | --- |
| 第 3 个画面 | 进入这个用例画布，顶部显示正在修改的画面 | 所有结果默认回到第 3 个画面 |
| 这一段发生什么 | 右侧解释剧情理解 | 关键图工具读取这段理解 |
| 改一下理解 | 打开自然语言编辑框 | 画面卡提示建议重新生成 |
| 人物当前样子 | 右侧显示林夏当前状态 | 关键图工具读取人物约束 |
| 保持一致 | 卡片状态变成已保持 | 后续关键图和视频生成都带上这个约束 |
| 还缺一个素材 | 右侧显示缺失的伤痕参考 | 关键图工具提示参考不足 |
| 生成参考 | 生成伤痕/半身参考 | 素材卡从缺失变成有候选 |
| 关键图生成工具 | 右侧显示它读取哪些卡、生成什么 | 不修改当前画面 |
| 生成关键图 | 工具进入生成中 | 完成后更新关键图候选卡 |
| 关键图候选 | 右侧显示大图和候选状态 | 图生视频工具仍等待采用图 |
| 采用这张 | 时间线缩略图更新 | 图生视频工具变为可运行 |
| 图生视频工具 | 右侧显示读取已采用关键图和动作说明 | 不修改当前画面 |
| 生成视频 | 工具进入生成中 | 完成后更新视频候选卡 |
| 生成的视频候选 | 右侧播放候选视频 | 当前画面还没被替换 |
| 用这个视频 | 第 3 个画面切换成视频 | 视频卡状态变为已用 |

### 11.5 UI 预览中的原型布局

```text
左侧：用例目标和步骤
中间：卡片 + 工具关系图
右侧：当前点击点的说明
```

中间关系图：

```text
这一段发生什么 ───────┐
                       ├──> 关键图生成工具 ──> 关键图候选 ──采用──┐
人物当前样子 ─────────┤                                           │
                       │                                           v
还缺一个素材 ─────────┘                                      当前预演画面
                                                                   │
预演里的一个画面 ─────────────────> 图生视频工具 ──> 视频候选 ──使用──┘
```

这个原型的关键不是做复杂节点编排，而是让用户理解：

```text
这些卡片提供生成所需的信息。
工具只是帮我生成候选。
只有我点采用/使用，预演才会真正更新。
```

以下概念只服务系统实现和团队协作，默认不暴露给普通用户：

| 内部概念 | 用户语言 |
| --- | --- |
| ScriptSection | 剧本段 |
| Situation | 这一段发生什么 |
| StoryboardLine | 分镜描述 |
| ContentUnit | 预演里的一个画面 |
| Keyframe | 关键图 |
| CreativeReference | 角色、地点、道具、风格 |
| CreativeReferenceState | 人物当前样子 |
| AssetRequirement | 还缺一个素材 |
| ContentVersion | 生成的视频候选 |
| WorkItem | 正在生成 / 待处理 |

## 12. 结论

MovScript V2 的产品方向应该是：

```text
用清晰的底层模型保证生产闭环，
用更少的用户概念承载创作体验。
```

画布卡片不是为了把所有对象可视化，而是为了让用户在具体创作对象上做决定。

最重要的不是“支持多少种卡片”，而是每张卡片是否能让用户马上知道：

```text
我现在该确认什么？
我还缺什么？
我点这个按钮之后结果会去哪里？
```
